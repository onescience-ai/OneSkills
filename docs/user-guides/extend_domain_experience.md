# 扩展你的科研领域经验

本指南面向想把自己的科研领域经验接入 `OneSkills` 的用户或贡献者。

目标不是让你“复制一套新 skill”，而是让你在不破坏产品设计的前提下，把领域知识接入现有能力体系。

## 先理解 OneSkills 的基本结构

`OneSkills` 默认遵循这条主链：

`onescience-workflow -> onescience-role -> onescience-skill -> execution skills`

这意味着：

- 领域差异，优先在 `workflow` 层表达
- 角色差异，优先在 `role` 层表达
- 执行实现，优先复用已有技能
- 只有真正新增了稳定执行能力时，才考虑新 skill

## 哪些内容适合加入

适合加入的内容包括：

- 某个科研领域的任务画像
- 常见的数据类型、变量、评估目标
- 该领域常见的角色分工
- 可复用的数据读取模板或模型改造模板
- 该领域的典型异常处理方式

## 哪些内容不建议直接加入

以下内容不建议直接作为新能力进入仓库：

- 单个项目的临时经验
- 只适用于一个团队的局部命名习惯
- 只适用于某个 agent 的提示偏好
- 与现有 skill 几乎重复，只是换了领域名的内容

## 你应该把内容加到哪里

### 情况 1：你想补充“领域理解”

如果你的经验主要帮助系统更好理解：

- 这个领域通常做什么
- 用户会怎么表达需求
- 常见任务阶段是什么

优先考虑补充：

- `skills/onescience-workflow/references/domain_profiles.md`
- `skills/onescience-workflow/references/workflow_matrix.md`

### 情况 2：你想补充“科研角色协作”

如果你的经验主要帮助系统更好判断：

- 谁先做
- 谁后做
- 中间要交接什么

优先考虑补充：

- `skills/onescience-role/references/role_matrix.md`

### 情况 3：你想补充“代码或模板经验”

如果你的经验主要是：

- 数据集读取模板
- 配置字段说明
- 模型改造套路
- 训练/推理脚本模板

优先考虑补充：

- `skills/onescience-coder/assets/`
- `skills/onescience-coder/references/`

### 情况 4：你觉得需要新增 skill

请先自查：

- 现有 skill 是否真的承接不了
- 这是不是稳定复用能力，而不是一次性经验
- 它是否拥有独立输入/输出/边界

如果以上三点不能明确回答，通常还不应该新增 skill。

## 推荐提交流程

1. 先写清你的领域经验解决什么问题
2. 判断它属于 `workflow`、`role`、`skill` 还是执行资产
3. 先做最小改动，不要一次铺开很多目录
4. 补一个对应的 QA case，或补充评测说明
5. 再提交 README 或说明文档更新

## 推荐自检问题

- 这条经验是通用的还是单项目临时的？
- 它是在补领域理解，还是在复制执行能力？
- 它是否破坏了现有主链？
- 用户真的会从中受益吗？
- 后续是否容易维护？

## 一句话原则

优先扩展现有分层，尽量不要复制 skill；优先补领域画像与模板资产，只有新增稳定执行能力时才新增 skill。
