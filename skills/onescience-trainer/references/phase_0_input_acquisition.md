# 阶段 0：输入获取

目标：把文本、URL、文件路径或上游 handoff 中分散的训练说明整理成有来源、可追踪的训练请求。

## 阶段资源评估

进入本阶段前，必须先使用 `workflow_contract.md` 中定义的初始 `type=resource` 召回结果，并围绕“输入获取”目标判断资源是否足够支撑训练请求规范化。

- 从已召回资源中筛选能解释训练说明来源、模型/数据/checkpoint 标识、训练入口、配置入口和领域背景的资源。
- 若缺少对用户文本、URL、模型名、数据名、checkpoint 或训练入口的解释，按本阶段目标再次调用 `type=resource` 补充资源。
- 补充召回应要求返回训练说明来源、资源适用范围、使用限制、字段含义和缺失项。
- 不得沿资源 `path` 直接读取资源文件来补洞；只能使用资源技能返回的 `content`、用户明确提供内容或上游已展开内容。
- 在 `source_capture.md` 中记录本阶段使用资源、暂不使用资源、补召回资源和仍缺失资源。

## 来源顺序

1. `step_handoff.inputs.instruction_source` 中的文本、URL、路径和注释
2. 初始与阶段补召回的 `type=resource` 资源内容
3. 用户提供的本地文件、项目路径、README、配置和训练脚本说明
4. 项目内已有训练入口、配置目录、checkpoint 元数据和日志说明
5. 官方网页、文档页面、模型页面或数据页面
6. 仅在一手来源不足时，使用论文或第三方说明

当来源来自在线页面或可能随时间变化时，依赖之前必须确认当前内容。

## 规范化要求

将训练来源整理为 `trainer_workdir` 下的 `source_capture.md` 与 `training_request.json`。其中：

- `source_capture.md` 记录已确认事实、证据位置、引用 URL、路径、缺口和假设
- `training_request.json` 记录结构化请求和后续阶段要读取的最小字段

如果输入是 URL：

- 读取网页当前内容，不要只根据 URL 名称猜测
- 抽取模型代码路径、训练入口、checkpoint、数据、optimizer、scheduler、batch 语义、训练命令、输出目录、依赖和评估要求
- 无法确认的事实使用 `MISSING:` 标记
- 只有为了继续规划必须先占位时，才使用 `ASSUMPTION:`

如果输入是自由文本：

- 保留原始关键信息，不要把“继续训练”“微调”“从头训练”混淆
- 区分模型代码路径、训练入口路径、配置路径和 checkpoint 路径
- 区分 train/val/test 数据来源和用户只举例说明的数据来源

## 抽取字段

在 `source_capture.md` 中至少记录：

- 训练目标：任务类型、目标变量、输出形式
- 模型入口：代码目录、模块、类、训练脚本或命令
- 权重语义：初始化权重、继续训练权重、微调权重、是否 resume optimizer state
- 数据信息：来源、格式、split、变量、样本组织、单位、标签来源
- 训练策略线索：optimizer、scheduler、epoch/step、batch、precision、分布式方式、评估频率
- 运行线索：依赖、硬件、日志目录、checkpoint 目录、执行方式
- 缺口和阻塞：哪些信息缺失会阻塞 codegen 或执行

## 决策门

在至少明确以下信息之前，不要进入代码生成：

- 训练是从头训练、继续训练还是微调
- 模型训练入口或足以生成训练入口的模型代码路径
- 最小训练数据来源与任务目标
- 至少一个训练策略核心约束，例如 optimizer、loss、batch 语义或用户接受的 `MISSING:` / `ASSUMPTION:` 范围

如果训练说明严重不足，返回阻塞状态，并要求用户补充文本、URL、训练脚本、checkpoint 路径或数据说明。
