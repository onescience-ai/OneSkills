# 静态需求一致性检查

本文档定义 `onescience-coder` 在最终验证进入静态检查分支时必须执行的检查。该检查只审查实现是否符合需求，不验证代码能否运行。

## 适用条件

仅在以下情况进入本分支：
- 本地环境不支持最小冒烟测试。
- 当前任务缺少本地运行所需的最小前提，无法形成可信的冒烟测试入口。
- 用户当前任务只允许执行静态一致性检查。

## 禁止项

- 不运行训练、推理、测试、benchmark 或示例脚本。
- 不下载数据、模型权重或依赖。
- 不提交远程作业。
- 不把报错日志调试当作本检查的一部分。

## 检查输入

整理以下材料形成 checklist：
- 用户原始需求。
- 用户确认的步骤执行信息。
- `type=resource` 技能返回的资源候选、摘要、路径和限制。
- 已读取的实现规格、契约、模板或工具说明。
- 用户指定目录、文件名、类名、函数名、参数和输出格式。
- `coding_conventions.md` 中的编码规范。

## 检查清单

逐项核对生成代码：

1. 文件落点是否符合用户指定目录结构；未指定时是否符合已确认目录规划。
2. 新增/修改文件是否只覆盖当前确认步骤的范围。
3. 类名、函数名、注册名、配置名、导入路径是否与确认内容一致。
4. 构造参数、配置字段、默认值、必填项是否完整。
5. 输入协议是否符合需求和资源契约。
6. 输出协议是否符合需求与约定产物格式。
7. shape、rank、维度名、batch 结构、时间步、变量通道、坐标/边/网格字段等结构化表示是否与确认内容一致；若任务不涉及此类结构，则按实际接口和数据流一致性核查。
8. 跨模块参数语义契约是否一致，包括类 / 实例 / 工厂 / callable / 配置 / 标识符等角色语义。
9. 是否避免非法 rank 或结构直连；若存在转换，是否有明确合法性依据。
10. 数据处理是否覆盖已确认的 split、normalization、collate、adapter/view、mask、padding 等要求。
11. 核心结构是否采用已确认的实现规格或组件契约。
12. 资源是否与目标实现方案一致；若资源只适用于其他方案，是否明确标记为不兼容或仅背景参考。
13. 输入输出语义是否完整对齐，包括 tuple arity、返回顺序、分支语义和字段命名。
14. 当前步骤承诺覆盖的训练、推理、评估、rollout、loss、metric、report 等入口是否完整。
15. 是否遵守最小改动路径，未擅自修改无关主循环、核心结构或资源文件。
16. 是否正确处理资源限制和 fallback，不把仅用于契约参考的资源当成已注册可运行入口。
17. 编码规范——OneScience 包依赖：是否符合 `coding_conventions.md` 的依赖规范。
18. 编码规范——数据路径处理：数据路径与输出路径是否通过函数参数、命令行参数或配置文件获取。

第 17、18 项为编码规范合规检查，必须逐项给出结论。

## 输出格式

```md
## 静态需求一致性检查结果
- 检查范围：
- 依据：
- 通过项：
- 参数语义契约合规（类 / 实例 / 工厂 / callable / 配置 / 标识符）：通过 / 不通过，已回改
- 编码规范合规（OneScience 包依赖 / 数据路径处理）：通过 / 不通过，已回改
- 已修正偏差：
- 需用户确认：
- 结论：通过 / 部分通过，需确认 / 未通过，已回改
```

## 发现偏差时的处理

1. 先修改代码。
2. 重新执行静态需求一致性检查。
3. 最终只输出修正后的实现摘要和最新检查结果。

## 无法确认时的处理

- 标为 `需用户确认`。
- 说明缺失信息。
- 不要将该项写成通过。
