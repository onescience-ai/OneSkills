# 分步编码工作流

本文档是 `onescience-coder` 的唯一详细执行流程文档。智能体应按本文档完成资源获取、步骤规划、执行信息输出、确认后编码、下一步交接与最终验证。

## 1. 总体原则

- 先取资源，再规划，再分步执行；不要直接开始写代码。
- 每个步骤都必须先输出执行信息，等待用户确认后再执行。
- 每个步骤只实现当前已确认内容，不顺手扩展后续步骤。
- 编码依据只能来自资源技能返回的 `content` 与用户明确提供的可用内容，不能沿资源 `path` 直接读文件。
- 优先复用已有实现与适配路径；只有复用不足时，才做最小必要改造。
- 若资源不足、契约冲突或关键决策未确认，停止并补资源或询问用户，不猜测实现。
- 所有步骤完成后，统一进入最终验证：本地可冒烟则冒烟，否则静态检查。
- 冒烟测试只允许在当前环境已经具备最小运行条件时执行；不得为了冒烟测试安装 conda 环境、创建新环境或安装额外依赖包。

## 2. 强制资源获取

### 2.1 接收任务

输入来自 orchestrator 或用户：

```yaml
coding_handoff:
  task_goal: <编码任务目标>
  reference_resources:
    - path: <资源路径>
      type: <资源类型>
      content: <资源内容摘要或完整内容>
```

### 2.2 立即调用资源技能

接收任务后，必须立即调用 `type=resource` 技能获取：
- 规格知识
- 使用知识
- 规划决策知识

无论上游是否已提供 `reference_resources`，都不能跳过这一步。

### 2.3 资源调用规则

- 查询中应带上用户需求，以及从 `reference_resources` 中提取的资源名称和关键概念。
- 资源不足时，可在步骤级再次调用 `type=resource`。
- 资源 `path` 仅用于标识，不得直接读取对应文件。

输入格式：

```yaml
resource_retrieval_request:
  user_request: <用户实现需求或当前步骤需求>
  task_state_summary: <当前任务状态摘要>
  content_request: "规格知识、使用知识和规划决策知识"
  filters:
    domain: <领域过滤，可选>
    keyword: <关键词过滤，可选>
```

输出格式：

```yaml
resource_retrieval_result:
  status: success | partial | failed
  matched_resources:
    - type: <具体资源类型>
      path: <资源路径>
      name: <资源名称>
      why_matched: <匹配理由>
      limitations: <使用限制>
      content: <完整的结构化内容或文本>
```

## 3. 初始资源筛选

把上游 `reference_resources` 与本轮 `matched_resources` 合并为初始资源清单，并完成一次筛选：

1. 识别任务核心目标。
2. 为每个资源判断与当前目标的关系。
3. 将资源标记为：
   - 核心资源：当前任务必须使用
   - 参考资源：可能有用，按步骤使用
   - 无关资源：当前不使用
4. 说明为什么选择某个资源、为什么不使用另一个资源。
5. 形成资源使用清单，并标记每类资源预计在哪些步骤中使用。

## 4. 目录结构与步骤规划

### 4.1 目录结构规划

- 用户指定目录或文件时，优先遵守。
- 用户只要求单文件时，可跳过目录规划。
- 未指定时，根据资源内容和任务目标规划最小完整结构。
- 不套用固定模板，不过度设计。

输出时应明确：
- 每个目录/文件的职责
- 哪些文件新增，哪些修改，哪些保持不变
- 文件之间的依赖关系

### 4.2 步骤定义与依赖识别

一个“步骤”应满足以下要求：
- 有清晰目标
- 有明确输入/输出或接口边界
- 可以独立确认
- 完成后能为后续步骤提供依赖

不要把整个任务当成单一步骤，也不要把没有独立价值的微小编辑拆成单独步骤。

步骤规划时要明确：
- 哪些步骤独立
- 哪些步骤依赖前置输出
- 正确执行顺序
- 如果用户指定顺序与依赖冲突，应指出冲突并说明建议顺序

## 5. 每步执行前的执行信息

每个步骤开始写代码前，必须先做步骤级资源评估，再输出执行信息。

### 5.1 步骤级资源评估

对当前步骤：
1. 识别本步骤需要的知识、接口、工具与实现依据。
2. 从初始资源清单中筛选出本步骤相关资源。
3. 说明每个资源在本步骤中的用途。
4. 判断当前资源是否足以支撑实现。
5. 若不足，则调用 `type=resource` 补充资源后再继续。

### 5.2 执行信息模板

资源就绪后，输出如下结构等待用户确认：

```md
【当前编码步骤】
- 步骤编号：
- 步骤名称：
- 前置依赖：
- 本步骤目标：

【资源依据】
- 本步骤使用资源：
- 筛选理由：
- 暂不使用资源：
- 本步骤补充资源：

【详细执行信息】
- 任务类别（data / model / component / config）：
- 复用判断：
- 最小改造路径：
- 输入定义：
- 输出定义：
- 变量组织方式：
- 保持不变的部分：
- 计划新增文件：
- 计划修改文件：
- 推荐复用的组件 / datapipe / model / tool：
- 主干结构：
- 实现方法：
- 主要 shape 变化：
- 当前步骤风险点：
- 需要用户确认的关键点：

## 待确认摘要
- 本步骤将做什么：
- 当前步骤属于哪类任务：
- 复用哪些现有实现：
- 最小改造路径：
- 将写入哪些文件：
- 不会修改哪些部分：
- 后续依赖：

## 用户操作
请回复：
1. `确认，执行本步骤`
2. `修改后再执行：<修改意见>`
```

## 6. 用户确认后的执行规则

用户确认当前步骤后：

1. 开始写代码前，必须已读取 `coding_conventions.md`。
2. 只实现当前已确认步骤，不顺手实现后续步骤。
3. 严格按照确认的文件清单和接口实现。
4. 优先复用已有实现、组件与适配路径；只有复用不足时，才做最小必要改造。
5. 使用资源 `content` 作为实现依据；`path` 只用于标识，不用于直接读取资源文件。
6. 若发现资源命名、内容或契约与目标实现不一致，停止并询问，不要猜测。
7. 保持改动范围最小，只触碰当前步骤声明的文件。
8. 在没有运行证据前，不得声称“已验证通过”。

## 7. 下一步骤交接

当前步骤完成后，如果还有后续任务：

- 输出当前步骤完成摘要。
- 输出下一步骤的执行信息。
- 等待用户确认后再继续。

不要自动生成下一步骤代码，除非用户明确一次性确认了多个具体步骤。

## 8. 所有步骤完成后的最终验证

所有编码步骤完成后，必须进行一次最终验证。

### 8.1 本地冒烟测试分支

仅当当前本地环境可直接运行目标代码、且具备最小必要前提时，才进入本地冒烟测试分支。

执行规则：
1. 只做最小范围的冒烟测试，不扩展成完整训练、推理、benchmark 或远程提交流程。
2. 总尝试次数不超过 6 次，含首次。
3. 每次失败后，只有在已经识别到明确可修正偏差时，才允许回改后再次尝试。
4. 达到 6 次上限后仍失败，必须停止，不得标记为 `success`。
5. 最终结果必须记录：尝试次数、执行依据、使用的本地入口/命令、结论。

### 8.2 静态需求一致性检查分支

若本地环境不支持最小冒烟测试，则执行 `static_requirement_review.md` 定义的静态需求一致性检查。

执行规则：
1. 从用户原始需求、已确认的执行信息、上游 handoff 与资源内容中提取需求清单。
2. 逐条检查生成或修改的代码是否满足需求清单。
3. 若发现偏差，先修改代码，再重新执行静态检查。
4. 输出 `静态需求一致性检查结果`。

若未进入本地冒烟测试分支，不得声称已经完成运行验证。

## 9. 特殊场景

### 用户指定完整目录结构

完全遵守用户给定结构。若资源契约建议不同结构，只说明差异和影响，不擅自改目录。

### 用户指定要写入的文件

只写入用户指定文件和必要的直接依赖文件。需要额外文件时，先在执行信息中请求确认。

### 用户要求“一次性全部生成”

仍先给出目录结构、步骤划分和每步执行信息。只有当用户明确确认“一次性执行全部步骤”后，才可以连续编码；每步完成后仍要统一进入最终验证。

### Paper2Code

如果任务是论文复现但没有上游 `onescience-paper-repro` 的 `coder_task_description.md`，退回论文复现前处理；不要自行搜索、下载或参考官方/第三方代码仓库。
