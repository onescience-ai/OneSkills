# Second Round Guide

请先参考 `./workflow.md` 中定义的工作流执行任务。

当前阶段是**第二轮代码生成阶段**。

执行要求：

1. 不要重新输出整份需求分析
2. 直接基于已确认的“详细执行信息”继续实现
3. 优先复用项目本地安装的 `./onescience/` 源码快照中已有模块；该目录位于 agent 的 `oneskills/onescience/` 安装根，不要扫描开发机磁盘路径
4. 先读取相关 `../assets/models/` / `../assets/contracts/`
5. 如契约信息不足，再回到对应源码锚点补充确认
6. 只修改与当前任务直接相关的文件
7. 代码生成完成后执行静态需求一致性 review，逐条核对代码是否满足已确认需求；发现偏差时先回改代码，再输出最终结果

设备与测试要求：

8. 若生成测试代码、样例代码或运行验证代码，默认优先使用 GPU
9. 优先采用：
   - `device = torch.device("cuda" if torch.cuda.is_available() else "cpu")`
10. 再将模型和输入张量移动到 `device`
11. 若环境中没有 GPU，应自动回退到 CPU，而不是写死为仅支持 CUDA

静态 review 要求：

- review 是需求一致性审查，不是运行校验或 debug；除非用户明确要求，不要运行训练、推理、测试、下载数据或提交远程任务
- 若上游交接包含 `coder_static_review_required=true`，最终输出必须包含 `静态需求一致性 review 结果`，否则不算完成代码生成
- 从已确认的详细执行信息和读取过的契约中整理 checklist
- 核对参数数量、参数名、默认值、必填字段和配置入口是否完整
- 核对模型结构是否符合要求，例如要求 Swin Transformer 时，不能实现成其它 backbone
- 核对输入输出变量、时间步、shape、batch 格式、loss、rollout、metric 和文件落点是否符合需求
- 发现不一致时修改代码并重新 review；无法确认的项在最终输出中标为需用户确认

输出时应包含：

- 最终完整代码
- 复用了哪些 OneScience 组件
- 主要 shape 变化
- 新增桥接模块或新增调用层改动说明
- 静态需求一致性 review 结果
- 如有必要，补充最小验证建议

如果用户未额外指定保存路径，则默认将生成文件保存到对应的 `case` 目录下。

如果已确认的设计中仍然存在会导致代码无法实现的关键冲突，请先用一个简短的 `md` 区块指出问题；否则直接开始生成代码。
